Postmortem

Retry storms and duplicate ingestion in async ETL

Summary: an async extraction workflow produced duplicate raw records after upstream throttling triggered overlapping retries. This write-up captures the incident shape and the controls added afterward.

Impact

Raw landing volume spiked, downstream normalization work reprocessed duplicate payloads, and analyst-facing SLA metrics became temporarily inflated. No data loss occurred, but trust in daily reporting degraded until the pipeline was replayed with corrected controls.

Timeline

  • T0: upstream API starts returning intermittent rate-limit responses.
  • T0 + 5m: async workers continue retrying with overlapping request windows.
  • T0 + 20m: raw row counts exceed expected batch envelope.
  • T0 + 45m: downstream aggregates show abnormal ticket volume and queue spikes.
  • T0 + 70m: run is halted and replay analysis begins.

Root cause

The extraction layer retried failed requests but did not bind retries tightly enough to idempotent request boundaries. The system treated retried windows as valid new fetches, which allowed duplicate events to land before normalization could deduplicate them.

What changed

  1. Added stronger request identity and replay-safe deduplication keys at the raw landing boundary.
  2. Introduced exponential backoff with jitter instead of synchronized retry bursts.
  3. Added row-count anomaly thresholds to detect retry amplification earlier.
  4. Separated transport retry logging from business-event acceptance metrics.

Operational lessons

  • Retries are a resilience feature only when idempotency is explicit.
  • Raw landing should preserve enough metadata to reconstruct request intent during incident review.
  • Volume-based alerts need expected-envelope logic, not just hardcoded thresholds.

Follow-up actions

Future iterations should add distributed tracing, retry budget dashboards, and explicit dead-letter handling so extraction failures can be isolated without contaminating downstream analytics.
